# dexplain v0.3 — diff, GitHub Action, output enrichment

**Date:** 2026-07-18
**Status:** approved (brainstormed section-by-section in session)
**Theme:** close the verify loop (`diff`), distribute via CI (Action), and make the
build report visual and causal (timeline + cache-miss cascade).

## Goals

1. `dexplain diff <a> <b>` — compare two images and/or build traces: size/layer
   deltas, findings resolved vs introduced, timing/cache deltas. `--fail-on`
   gates on INTRODUCED findings, closing the fixRisk verify loop in CI.
2. `tomerab1/dexplain-action` — a dedicated-repo composite GitHub Action:
   gate via `--fail-on` + findings table in `$GITHUB_STEP_SUMMARY`.
3. Output enrichment — a terminal Gantt timeline of the build (parallel stages
   visible) and a `cache-miss-cascade` trace rule ("`COPY . .` broke the cache:
   8 downstream steps rebuilt, 47.3s"), plus downstream-count enrichment of the
   static cache-invalidation finding.

## Non-goals

- No MCP server (explicitly dropped by the user).
- No runtime-config diff (ENTRYPOINT/ENV/USER deltas) — deferred; cheap later add.
- No PR-comment mode in the Action (job summary only; permissions stay zero).
- No HTML/flamegraph output — the timeline is terminal-native (steps aren't
  nested, so a Gantt is the correct shape).
- Zero new npm dependencies in either repo.

## 1. `dexplain diff`

- **CLI:** `dexplain diff <a> <b>`; each side is an image ref OR a rawjson trace
  file (auto-detect: existing file ending `.ndjson` → trace, else image ref);
  `--trace-a <file>` / `--trace-b <file>` add timing to an image side. Docker
  status is checked only when at least one side is an image (the NEEDS_DOCKER
  gate becomes per-invocation for this command). Sides are collected with the
  existing collectors (`collectImage`, `parseBuildTrace`) — no new Docker calls.
- **Deltas — one pure module each under `lib/diff/`:**
  - `layers.mjs`: layers matched across sides by `createdBy` text (multiset,
    duplicates pair in order). Matched → byte delta; unmatched A → removed;
    unmatched B → added. Headline: total size delta.
  - `findings.mjs`: run `runRules` per side on whatever inputs that side has
    (same `requires` filtering as today). Match findings by identity =
    `ruleId` + rule-supplied evidence key when present (`createdBy`, `name`,
    `packageManager`, …), fallback digits-normalized title. A-only = resolved,
    B-only = introduced.
  - `timing.mjs` (when both traces exist): steps matched by command text;
    per-step duration deltas; cache transitions (hit→miss "cache lost",
    miss→hit "cache gained"); wall-clock delta headline.
- **Output:** human renderer sections `SIZE`, `LAYERS`, `FINDINGS`, `TIMING` —
  improvements green, regressions red; `--json` emits `{a, b, deltas}` with the
  existing finding shape. `--fail-on <sev>`: exit `EXIT.FINDINGS` (5) only when
  an introduced finding is at/above the severity.
- **Tests:** fixture pairs (two history/inspect fixtures; the existing
  cached/uncached rawjson fixtures diff against each other), renderer snapshot
  tests, one live end-to-end comparing two builds of a demo Dockerfile.

## 2. GitHub Action (`tomerab1/dexplain-action`, dedicated public repo)

- **Composite action** (no container): runs on the runner's Node (≥20 on hosted
  runners); Docker needed only for `command: build|image`.
- **Inputs:** `command` (dockerfile|build|image, default dockerfile), `target`
  (default Dockerfile), `args` (build passthrough), `fail-on`
  (high|medium|low|none, default high), `version` (npm version, default latest;
  README recommends pinning).
- **Steps:** (1) `npx -y @tomerab1/dexplain@<version> <command> <target> [args]
  --json-out report.json --no-color`, exit code captured, not yet propagated;
  (2) vendored `summary.mjs` renders report.json → `$GITHUB_STEP_SUMMARY`
  (headline stats + findings table: severity, rule, location, title, fixRisk);
  (3) exit with the captured code — summary always renders before the gate fires.
- **Repo contents:** `action.yml` (with Marketplace branding), `summary.mjs`,
  README (usage snippets incl. `uses: tomerab1/dexplain-action@v1`), MIT
  LICENSE, dogfood workflow (fixture Dockerfile: one `fail-on: none` job
  asserting the summary/report, one job asserting the exit gate fails).
- **Versioning:** tag `v0.1.0` + moving `v1` tag. Marketplace listing = manual
  user step on the first release.

## 3. Output enrichment (dexplain repo)

- **Timeline** (`lib/render/timeline.mjs`): terminal Gantt after the headline —
  one row per step (truncated command), bars scaled to wall clock and terminal
  width (capped), offsets from `startedMs`, cached steps dim, internal steps
  (export) included so the tail is visible; multi-stage parallelism appears as
  overlapping bars. Shown when the build exceeds duration + step-count floors
  (new `THRESHOLDS` entries); `--timeline` / `--no-timeline` override.
- **`cache-miss-cascade`** (new trace rule): per stage, the first uncached step
  with uncached successors gets a finding — "`<cmd>` (step N) broke the cache:
  K downstream steps rebuilt, Xs". `estimatedImpact` = summed downstream
  durations. Noise guard: fires only when at least one step BEFORE the miss was
  cached (first-ever builds have no cascade story).
- **cache-invalidation enrichment:** static finding detail gains "invalidates N
  later instructions in this stage" from the AST.

## Execution plan and gates

Waves (haiku agents, file-disjoint):
1. **Wave A (parallel):** Action repo scaffold (own repo, agent uses `gh` to
   create + push under tomerab1) ∥ `diff` delta modules + fixtures
   (lib/diff/*, test/diff/*) — pure modules first, no cli wiring.
2. **Wave B:** diff cli wiring + human/json rendering (cli.mjs, render) + live
   end-to-end.
3. **Wave C:** enrichment — timeline renderer ∥ cache-miss-cascade rule (+
   cache-invalidation detail), then cli/README/SKILL/CHANGELOG docs.
4. Gates at every wave: full `node --test` green, aislop scan 100/100,
   thematic batch commits pushed per wave. Version 0.3.0 bump + CHANGELOG at
   the end; `npm publish` and the Marketplace release checkbox stay manual
   user steps.
